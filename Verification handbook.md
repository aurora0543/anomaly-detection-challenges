## 计算开销维度挑战测试方案

### 端侧资源受限 — 功耗墙与无风扇散热约束下的实时推理矛盾

1. 测试方法：**恒温箱压力测试法**

2. 测试指标

| 指标 | 测量方式 | 验证标准 |
|------|----------|----------|
| **推理延迟 (ms)** | 单帧端到端时间（预处理→推理→后处理） | 布匹产线速度要求：通常 ≥ 30 FPS (≤33ms/帧) |
| **持续吞吐量 (FPS)** | 30分钟连续推理的平均帧率 | 不低于峰值帧率的70%（降频阈值） |
| **功耗 (W)** | 功率计实时采样 | 无风扇设备 ≤ 10W 持续运行 |
| **芯片温度 (°C)** | 内置温度传感器 | 不超过降频阈值（通常85°C） |
| **降频发生时间** | 从满负载开始到首次降频的时长 | 无风扇场景 < 5分钟即触发降频 |

3. 测量流程
   - 温度控制 -> 启动推理，固定图像分辨率 -> 记录（延迟、帧率、功耗、温度，采样频率）
   - 持续30min，观察温度曲线与帧率变化

### 注意力机制高复杂度 — 高分辨率实时吞吐率冲突

1. 测试方法：**分辨率-吞吐量扫频测试**

2. 测试指标

| 指标 | 测量方式 | 验证标准 |
|------|----------|----------|
| **FLOPs** | 理论计算量（thop/profile工具） | 注意力复杂度 O(n²) vs 卷积 O(n) |
| **实际吞吐量 (FPS)** | 不同分辨率下的实测帧率 | 分辨率翻倍，吞吐量下降幅度 |
| **显存带宽占用率** | nvidia-smi dmon 监控 | 高分辨率下是否带宽瓶颈 |
| **推理时间分解** | 各层耗时占比（PyTorch Profiler） | 注意力层耗时占比随分辨率变化 |

3. 测试步骤
   1. 准备分辨率梯度：256 / 512 / 1024 / 2048
   2. 对每个模型在各分辨率下运行 100 次 warmup + 1000 次正式测试
   3. 记录：单帧推理时间、各阶段（预处理/骨干/颈部/头部）耗时
   4. 绘制 **分辨率-吞吐量曲线**，计算下降斜率
   5. 纯卷积模型（YOLO）vs 含注意力模型的曲线斜率差异

---

### 训练阶段硬件门槛与能耗 — 换产引发云端重训延迟与边缘增量学习局限

1. 测试方法：**换产场景模拟训练测试**

2. 测试指标

| 指标 | 测量方式 | 验证标准 |
|------|----------|----------|
| **全量训练时间 (h)** | 从随机初始化到收敛 | 换产周期（通常数小时到数天） |
| **增量训练时间 (min)** | 基于预训练的微调 | 边缘设备能否在可接受时间内完成 |
| **训练能耗 (kWh)** | 功率 × 时间 | 云端训练的碳足迹与成本 |
| **GPU显存峰值 (GB)** | 训练过程峰值占用 | 是否需要多卡/高端GPU |
| **边缘增量学习可行性** | 边缘设备上训练的收敛速度与精度 | 边缘能否独立完成换产适配 |

3. 测试场景设计

| 场景 | 描述 | 模拟换产类型 |
|------|------|--------------|
| **场景A：全量换产** | 全新布匹纹理，从零训练 | 全新面料品类导入 |
| **场景B：增量微调** | 基础模型 + 10%新数据微调 | 同品类花色变化 |
| **场景C：Few-shot适配** | 仅5-20张新样本适配 | 小批量定制订单 |
| **场景D：边缘增量** | 在 Jetson 上完成增量训练 | 无云端连接的离线场景 |

4. 测试步骤

   1. 准备基础布匹数据集 + 3种"新面料"子集（各50/200/1000张）

   2. 分别在云端（A100）和边缘（Jetson）执行四种场景

   3. 记录：训练时间、最终mAP/AUROC、峰值显存、能耗估算

   4. 对比：

      - 全量训练 vs 增量学习的时间差（验证换产延迟）

      - 云端 vs 边缘的训练效率比（验证边缘局限）

      - 样本量与性能提升的边际曲线

### 扩展性瓶颈 — 多验布机并发带来的部署成本与工程复杂度激增

1. 测试方法：**多实例并发压力测试**

2. 测试指标

| 指标 | 测量方式 | 验证标准 |
|------|----------|----------|
| **单GPU承载路数** | 单卡上并发推理路数与单路延迟 | 每增加一路，延迟上升幅度 |
| **并发吞吐总量** | N路并发的总FPS | 是否线性增长 / 何时饱和 |
| **CPU占用率** | 多进程并发的CPU负载 | 预处理是否成为瓶颈 |
| **部署成本/路** | 硬件成本 ÷ 最大路数 | 单位验布机的AI算力成本 |

3. 测试步骤
   1. 在单台 GPU 服务器上启动 N 个推理实例（N=1,2,4,8,16,32）
   2. 每个实例模拟一路验布机的图像输入流
   3. 记录每路的平均延迟、总吞吐量、GPU利用率
   4. 绘制 **并发数-延迟曲线** 和 **并发数-总吞吐曲线**
   5. 找到"延迟翻倍点"和"吞吐饱和点"

#### 关键对比
| 模型 | 单卡最大路数 (延迟≤33ms) | 单路成本估算 | 验证点 |
|------|--------------------------|--------------|--------|
| YOLOv8n | ? | 低 | 高密度部署可行性 |
| YOLOv11 | ? | 中 | 精度换成本的trade-off |
| PatchCore | ? | 高 | memory bank对并发的影响 |
| MoECLIP | ? | 高 | 专家路由的并发开销 |

### 数据分布异构性 — 跨机台概念漂移与中心聚合模型性能退化

#### 测试方法
**跨机台泛化性能测试**

#### 测试指标
| 指标 | 测量方式 | 验证标准 |
|------|----------|----------|
| **单机台精度** | 在A机台数据上训练，A机台测试 | 基准性能 |
| **跨机台精度** | A机台训练，B机台测试 | 性能下降幅度 |
| **联邦聚合精度** | 多机台联邦训练后，全局模型精度 | 是否优于单机台模型 |
| **漂移检测延迟** | 从分布变化到检测出漂移的时间 | 概念漂移的响应速度 |

#### 测试设计
1. 准备 3-5 个"机台"的数据子集（通过光照、角度、噪声模拟不同机台特性）
2. 训练方案对比：
   - **方案A**：单中心模型（用所有数据集中训练）
   - **方案B**：各机台独立模型
   - **方案C**：联邦学习聚合模型（FedAvg/FedProx）
3. 测试各方案在"本机台"和"跨机台"上的性能
4. 模拟概念漂移：逐步引入新纹理/新缺陷，观察性能衰减曲线

## 内存开销维度测试方案

### HBM容量与带宽瓶颈 — 图像高速采集与模型推理对总线带宽的竞争

#### 测试方法
**内存带宽压力测试**

#### 测试指标
| 指标 | 测量方式 | 验证标准 |
|------|----------|----------|
| **模型显存占用 (MB)** | 推理时峰值显存 | 是否超出边缘设备显存 |
| **显存带宽占用 (GB/s)** | nvidia-smi dmon -s m | 带宽利用率是否饱和 |
| **数据拷贝耗时占比** | PCIe/H2D/D2H 传输时间 | 数据搬运是否成为瓶颈 |
| **采集+推理并发性能** | 图像采集线程与推理线程并行时的帧率下降 | 总线带宽竞争程度 |

#### 测试步骤
1. **基线测试**：纯推理（数据已在显存中）的吞吐量
2. **真实场景测试**：图像从磁盘/摄像头 → CPU内存 → GPU显存 → 推理，全流程
3. 对比两者吞吐量差异，计算"数据搬运开销占比"
4. 使用 `nvidia-smi dmon` 监控显存带宽利用率
5. 模拟高速采集场景：4K@60fps 图像输入，观察推理帧率下降

#### 关键对比实验
| 实验 | 变量 | 验证假设 |
|------|------|----------|
| 256px vs 2048px | 图像大小 | 高分辨率下带宽瓶颈更显著 |
| 批处理=1 vs 批处理=8 | batch size | 批处理能否摊薄带宽开销 |
| 统一内存 vs 显式拷贝 | 内存模式 | Jetson 统一内存架构的优劣 |

### 存储层次约束 — 低位宽量化对微弱缺陷纹理特征的抹除效应

#### 测试方法
**量化精度损失对照实验**

#### 测试指标
| 指标 | 测量方式 | 验证标准 |
|------|----------|----------|
| **模型体积 (MB)** | FP32 / FP16 / INT8 / INT4 各精度下大小 | 压缩比 |
| **推理速度提升** | 各精度下的FPS比 | 量化加速比 |
| **整体精度 (mAP/AUROC)** | 量化前后的检测精度变化 | 总体精度损失 |
| **微缺陷检出率** | 针对 <10px 缺陷的召回率 | 微弱特征是否被抹除 |
| **缺陷定位精度** | 缺陷边界IoU变化 | 纹理细节是否模糊 |

#### 测试矩阵
| 模型 | FP32 | FP16 | INT8 (PTQ) | INT8 (QAT) | INT4 |
|------|------|------|------------|------------|------|
| YOLOv8n | ✓ 基准 | | | | |
| YOLOv8s | ✓ 基准 | | | | |
| PatchCore | ✓ 基准 | | | | |
| EfficientAD | ✓ 基准 | | | | |
| MoECLIP | ✓ 基准 | | | | |

#### 测试步骤
1. 以 FP32 模型为基准，记录各精度指标
2. 使用 TensorRT / ONNX Runtime 导出 FP16、INT8 模型
3. INT8 校准：使用 100-500 张正常布匹图像作为校准集
4. 在测试集上评估各量化版本的性能
5. **重点分析**：按缺陷尺寸分层统计召回率变化
   - 微缺陷（<10px）：量化前后召回率差
   - 小缺陷（10-30px）：量化前后召回率差
   - 大缺陷（>30px）：量化前后召回率差

#### 量化策略对比
| 量化策略 | 描述 | 预期对微缺陷的影响 |
|----------|------|--------------------|
| **Per-Tensor** | 整个张量一个缩放因子 | 影响最大，微缺陷可能消失 |
| **Per-Channel** | 每个通道一个缩放因子 | 中等影响 |
| **混合精度** | 敏感层保留FP16，其余INT8 | 影响最小，但加速比降低 |
| **QAT训练感知量化** | 训练中模拟量化噪声 | 精度损失最小 |

## 通信开销维度测试方案

### 4.1 挑战1：分布式版本同步 — 模型更新下发的高效性与多机台版本同步

#### 测试方法
**模型分发与同步测试**

#### 测试指标
| 指标 | 测量方式 | 验证标准 |
|------|----------|----------|
| **模型更新包大小 (MB)** | 全量更新 vs 差分更新 | 传输数据量 |
| **单设备更新耗时 (s)** | 下载 + 校验 + 加载 | 单台设备升级时间 |
| **百台设备同步时间 (s)** | 100台设备全部完成更新的时间 | 大规模部署效率 |
| **网络带宽占用 (Mbps)** | 更新期间的峰值带宽 | 对产线网络的冲击 |
| **版本一致性** | 更新后各设备的模型版本号 | 是否存在版本分裂 |

#### 测试方案
1. **全量更新**：完整模型文件下发
2. **差分更新**：仅传输权重增量（Δ权重）
3. **结构化更新**：仅更新特定层（如检测头/分类头）
4. **联邦蒸馏更新**：仅传输知识蒸馏后的小模型

#### 测试场景
| 场景 | 网络条件 | 设备数 | 模拟环境 |
|------|----------|--------|----------|
| 理想LAN | 1Gbps 有线 | 10台 | 工厂内网 |
| 真实工厂 | 100Mbps 有线，有丢包 | 50台 | 典型车间 |
| 边缘恶劣 | 10Mbps 无线，高延迟 | 100台 | 分布式多厂区 |

#### 测试步骤
1. 搭建模拟网络环境（使用 tc 命令控制带宽/延迟/丢包）
2. 部署 N 个边缘节点 + 1 个中心服务器
3. 触发模型更新，记录各节点的：
   - 开始下载时间
   - 下载完成时间
   - 校验通过时间
   - 模型加载完成时间
4. 统计整体同步完成时间分布（P50/P95/P99）
5. 对比不同更新策略的带宽节省比和时间节省比

### 非对称数据流控 — 高危缺陷图像块的毫秒级异步回传

#### 测试方法
**事件驱动回传延迟测试**

#### 测试指标
| 指标 | 测量方式 | 验证标准 |
|------|----------|----------|
| **检测到回传延迟 (ms)** | 从模型检出缺陷到图像块到达云端的时间 | 高危缺陷的响应速度 |
| **回传数据量 (KB/事件)** | 单次缺陷回传的数据大小 | 带宽占用 |
| **正常帧零回传率** | 正常图像是否完全不上传 | 带宽节省效果 |
| **并发事件处理能力** | 1秒内10个缺陷同时发生的回传成功率 | 突发场景稳定性 |

#### 测试设计
1. **全量回传方案**：所有图像帧都上传云端（基线）
2. **事件触发方案**：仅检出缺陷时才回传图像块
3. **分级回传方案**：
   - 高危缺陷（断经、破洞）：立即回传完整图像 + 位置
   - 中危缺陷：回传裁剪后的缺陷区域
   - 低置信度异常：仅回传特征向量，人工复核时再取图

#### 测试步骤
1. 构建测试视频流：混入 1% 的缺陷帧（模拟真实产线缺陷率）
2. 边缘端运行检测模型，触发缺陷时启动回传
3. 时间戳埋点：
   - T1：模型输出缺陷检测结果
   - T2：图像块编码压缩完成
   - T3：网络发送开始
   - T4：云端接收完成
   - T5：云端解析入库完成
4. 计算各阶段延迟，绘制延迟分布
5. 统计 1 小时内的总回传数据量，与全量回传对比

#### 预期验证结果
- 事件触发回传：数据量减少 90%-99%（正常帧不上传）
- 单缺陷回传延迟：50-200ms（局域网），500-2000ms（广域网）
- 验证：非对称流控可大幅节省带宽，但需保证高危缺陷的毫秒级响应
- 异常检测模型因误报率较高，实际回传数据量可能高于预期

### 硬实时控制链协同 — 检测信号与织布机的硬实时协同

#### 测试方法
**端到端控制延迟测试**

#### 测试指标
| 指标 | 测量方式 | 验证标准 |
|------|----------|----------|
| **检测-执行总延迟 (ms)** | 缺陷出现 → 检测 → 信号输出 → 织机响应 | 硬实时要求（通常 < 10ms） |
| **延迟抖动 (μs)** | 延迟的标准差/最大值 | 确定性/可预测性 |
| **丢包率/误码率** | 控制信号的可靠性 | 工业级可靠性要求 |
| **优先级抢占能力** | 高优先级缺陷能否打断低优先级任务 | 实时操作系统特性 |

#### 测试方案
这部分需要硬件在环（HIL）测试，软件模拟方案：

1. **模拟织机控制信号**：使用 GPIO / 工业总线（Modbus/CAN）输出
2. **时间戳同步**：使用 PTP 精确时间协议同步边缘端与控制器
3. **延迟分解**：
   - 图像采集延迟（曝光+读出）
   - 推理延迟（预处理+推理+后处理）
   - 决策延迟（缺陷等级判定+逻辑判断）
   - 通信延迟（信号传输到PLC）
   - 执行延迟（PLC扫描+执行机构响应）

#### 测试步骤
1. 搭建模拟环境：图像采集卡 → 边缘推理 → PLC模拟器
2. 使用高精度计时器（μs级）在各节点打时间戳
3. 运行 1000 次缺陷检测-控制输出循环
4. 统计延迟分布：最小值、平均值、最大值、P99
5. 对比：
   - 纯软件方案（Linux + 应用层）
   - 实时内核方案（PREEMPT_RT）
   - 硬件加速方案（FPGA/ASIC 推理 + 硬实时总线）

### 5.1 测试环境搭建步骤

```
Step 1: 数据集准备
  ├─ 下载/整理华纺数据集
  ├─ 按缺陷类型、尺寸分级标注

Step 2: 模型训练与导出
  ├─ 监督路线：YOLOv8n/v8s/v11 各训练3个epoch
  ├─ 异常路线：PatchCore/RD4AD/EfficientAD 各训练完成
  ├─ MoECLIP：配置专家数=8，Top-k=2
  └─ 导出各精度版本（FP32/FP16/INT8）

Step 3: 基准性能测试（计算+内存）
  ├─ 单模型推理延迟/吞吐测试
  ├─ 显存/内存占用测试
  ├─ 分辨率扫频测试
  └─ 并发路数压力测试

Step 4: 场景化测试
  ├─ 端侧功耗/温度压力测试
  ├─ 量化精度损失对比测试
  ├─ 跨机台泛化测试
  └─ 增量学习/换产测试

Step 5: 通信与分布式测试
  ├─ 模型更新同步测试
  ├─ 事件回传延迟测试
  └─ 多节点并发升级测试

Step 6: 数据汇总与分析
  ├─ 生成各维度对比表格
  ├─ 绘制性能曲线
  ├─ 验证各挑战点是否成立
  └─ 输出实测验证报告
```

### 5.2 数据记录模板

每个实验都需记录以下元数据：
- **硬件环境**：GPU型号、驱动版本、CUDA版本、CPU、内存
- **软件环境**：PyTorch版本、推理框架版本、操作系统
- **模型配置**：具体版本、输入分辨率、batch size、精度
- **测试数据**：数据集名称、样本数量、缺陷分布
- **重复次数**：每个实验重复 5 次，取均值和标准差
- **时间戳**：测试开始/结束时间

## 六、预期验证结论汇总

| 挑战维度 | 具体挑战 | 预期验证结论 | 关键量化指标 |
|----------|----------|--------------|--------------|
| **计算开销** | 端侧资源受限 | ✅ 成立：无风扇场景下降频严重，实时性难保障 | 温度>85°C时间 < 5min |
| | 注意力高复杂度 | ✅ 成立：高分辨率下吞吐率超线性下降 | 2048px时吞吐下降 > 20× |
| | 训练硬件门槛 | ✅ 成立：换产重训需数小时，边缘无法独立完成 | 边缘训练速度 < 云端 1/20 |
| | 扩展性瓶颈 | ✅ 成立：异常检测模型单卡承载路数极少 | PatchCore单卡 ≤ 4路 |
| | 数据异构性 | ✅ 成立：跨机台精度下降 10%-30% | 跨机台mAP下降 > 15% |
| | 运维开销 | ✅ 成立：百台设备升级需分钟到小时级 | 95%完成时间 > 15min |
| **内存开销** | HBM带宽瓶颈 | ✅ 成立：高分辨率下带宽饱和，数据搬运占比高 | 带宽利用率 > 80% |
| | 量化抹除效应 | ✅ 成立：INT8量化导致微缺陷召回率显著下降 | 微缺陷召回下降 > 10% |
| **通信开销** | 版本同步效率 | ✅ 成立：异常检测模型同步开销远高于监督模型 | PatchCore更新 > 10min |
| | 非对称流控 | ✅ 成立：事件触发可节省90%+带宽，但需保证高危响应 | 回传延迟 < 200ms |
| | 硬实时协同 | ✅ 成立：纯软件方案无法满足硬实时要求 | 总延迟 > 50ms，抖动大 |
